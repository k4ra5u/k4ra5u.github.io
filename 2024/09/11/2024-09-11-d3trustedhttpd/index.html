hint: The flaw of RPC between httpd and trusted_core makes the similarity of Face ID leaked. Can you forge a Face ID to bypass login?

A bad TA may cause memory corruption at CA. Please find vulnerabilities in secure file system (mainly about file-create, file-rename and file-info interfaces).

整体逻辑分析

服务端使用mini_httpd作为框架接收http请求
mini_httpd 与 名为op_tee_trusted_core的转发平台 模块使用RPC进行通信


所有涉及文件和用户的操作均通过TEEC_InvokeCommand函数并调用ioctl来发送给op_tee内核进行验证。
内核的解析模块在ae开头的ta模块里，并通过TEEC_InvokeCommand第二个参数来定位处理函数。

  1. mini_httpd监听端口，根据收到的请求类型（GET/HEAD/POST）、URL路由、参数等进入相应的逻辑（sub_AA24）

  2. 在密码登录和faceid登录的处理逻辑中，利用RPC和自定义命令格式（!! command authxxxxxx）通过Unix sock与optee_d3_trusted_core进行通信。

  3. 在optee_d3_trusted_core的处理逻辑中（sub_4027F0），对收到的命令进行解析，并调用TEEC_InvokeCommand与内核模块进行通信

在进行reset操作时，会把密码重新设置成弱口令password的sha256

所以通过disable->reset->enable操作可以将guest用户的密码重置为password。
httpd中对disable操作的执行逻辑：

首先发送kickout指令，然后发送disable指令。
kickout指令在op_tee内核中的操作：

在全局的session链表中寻找对应要删除的session的uid。如果匹配上就从链表中摘除，并free掉这个session

disable逻辑如下：

正常来说已经disabled的用户是不能用来登录的，将用户disable的流程中也首先把已经登陆的用户踢掉。所以不会出现某个session所属用户是disabled的情况。
然而如果利用条件竞争同时登陆用户和禁用用户呢，
首先执行kickout ，然后可能会先执行userlogin，再执行disable，这样就会得到一个被禁用的用户session。
这样会带来什么问题呢，首先看session的生成：

注意session的user_info此时指向全局的enable_user_list，disable操作后这个user_info的结构体从enable_user_list摘除，转到disable_user_list中。

在reset一个disabled的用户时：

会先把这个结构体移除并free掉session->user_info指向了freed addr。
之后申请的face_data的堆又会把刚刚free掉的空间申请回来并写入自己的face_data
在执行后面的一系列操作之后的会把当前已有session的user_info进行一次更新。
问题出现在更新上，if ( uid == session->user_info->uid )会造成UAF
因此只要通过了这个check保证user_info不会被覆盖，

step3

关注文件操作，op_tee维护了一个全局的sec_file_list列表
status判断状态，optee_core根据objectID去查找文件信息和文件内容。

文件信息存储在文件结构体中：

文件删除分2中方式，擦除模式会彻底删除该文件，并将status置为empty。
删除模式只将status置为delete，并不会改变文件内容。

第一个漏洞是在d3_core_create_secure_file中，当要插入的文件块是delete状态时，不会修改文件结构体中的file_status字段，与之相对的，如果是empty状态，则修改文件file_status字段为0xFFFF1000

第二个漏洞在d3_rename_sec_file中，文件名可以写满128个字节。注意到如果之间这个slot是一个被删除的文件夹，那么它的file_status为0xffff1001,结合第一个漏洞,此时 strlen_51C0(v7->file.filename)将会把filename、file_status和content连起来更新到v7->file.filename_size中。

在CA与TA的通信中，paramTypes定义了共享内存的方式，经查表，使用TEEC_MEMREF_TEMP_INPUT方式，即CA中存放的file_args_addr的指针会通过TEEC_MEMREF_TEMP_INPUT的方式传递给TA。

在d3_get_sec_file_info中， memmov会造成堆栈空间溢出。
